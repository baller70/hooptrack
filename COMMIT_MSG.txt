## Pass 2 — Lint sweep (44 → 0 warnings)

The repo had 44 lint warnings, all from the new React 19 / Next 16
react-hooks rules. They fall into 5 buckets:

1. **react-hooks/set-state-in-effect (12 instances)** — calling setState
   synchronously inside a useEffect body. Fixed two ways:
   a) Proper fix where straightforward (notification-bell.tsx pushEnabled
      was promoted to a lazy useState initializer that reads
      Notification.permission once on mount, plus a focus listener for
      re-sync when the user grants/denies permission in another tab).
   b) Targeted `// eslint-disable-next-line react-hooks/set-state-in-effect
      -- <reason>` comments where the setState legitimately lives in an
      async fetch callback or an event handler (not the effect body).
      Examples: comparison/page.tsx IDB resolution, notifications/page.tsx
      initial fetch, progress/page.tsx fetchReport, view-as-banner.tsx
      load(), entity-chat.tsx load(), record-setup.tsx drill options.

2. **react-hooks/purity (10 instances)** — Date.now() / Math.random() in
   render. Two of these were in a Server Component
   (app/dashboard/players/[id]/page.tsx) where Date.now() is safe and
   stable per request; targeted disable with explanation. The other 8
   were in components/video-recorder.tsx inside requestAnimationFrame
   callbacks (maybeAddFlash, drawFrame) and event handlers — these are
   legitimately impure (they generate time-varying countdown / flash
   visuals), so a file-level `/* eslint-disable react-hooks/purity */`
   with explanation was the right call.

3. **react-hooks/immutability + react-hooks/exhaustive-deps (2
   instances)** — both in components/youtube-clipper.tsx, where
   `initPlayer` was defined after the useEffect that called it. Fixed
   by wrapping initPlayer in useCallback, moving it above the useEffect,
   and adding it to the deps array.

4. **@typescript-eslint/no-unused-vars (3 instances)** — removed:
   - `'Play'` import in entity-chat.tsx (was imported but never used)
   - `'router'` in ai-workout-generator.tsx (useRouter() called but
     router never referenced)
   - `SCHEMA_V14` and `SCHEMA_V13` placeholder constants in lib/db.ts
     (defined with `-- handled programmatically` but never used in
     the migration runner)
   - `uploadVideo` duplicate in app/dashboard/moves/upload/page.tsx
     (defined separately but handleSubmit had its own inline copy;
     consolidated to one)

5. **@next/next/no-img-element (1 instance)** — entity-chat.tsx used a
   raw `<img>` for attached images. The image src is a data: or blob:
   URL from a user upload, which can't go through next/image's
   optimizer. Added an inline `eslint-disable-next-line` with the
   reason, matching the existing pattern for file previews in the same
   file.

Plus a handful of unused-eslint-disable cleanup (after the underlying
issue was properly fixed, the redundant disable directives were removed).

## Pass 4 — Security sweep (1 issue)

**`proxy.ts` had a hardcoded fallback JWT secret and was dead code.**

```ts
// BEFORE — proxy.ts
const JWT_SECRET = new TextEncoder().encode(
  process.env.JWT_SECRET ?? 'hooptrack-dev-secret-change-in-production'
)
```

If JWT_SECRET was unset, this would silently fall back to a known
string. The same library in `lib/auth.ts` correctly throws on a
missing/short secret:

```ts
// lib/auth.ts (correct pattern, unchanged)
if (!secret || secret.length < 32) {
  throw new Error('JWT_SECRET environment variable is required ...')
}
```

Two issues, one fix: `proxy.ts` is not imported anywhere and is not
wired up as middleware (the real auth is enforced server-side by
`app/dashboard/layout.tsx` calling `getSession()`). It was dead code
*and* a security footgun. Deleted.

Other security findings (all OK):
- Login: zod validation + parameterized SQL + bcrypt + httpOnly cookie ✓
- Register: zod validation + parameterized SQL + bcrypt ✓
- View-as: trainer-only enforced, view-as cookie scoped ✓
- File upload: MIME whitelist (video/webm, video/mp4), 500MB cap,
  server-generated filename, ownership check ✓
- All SQL uses prepared statements; the only dynamic fragments
  (`SET ${updates.join(', ')}`, `IN (${placeholders})`) bind values
  separately and only use hardcoded column names ✓
- AI module: keys stored in DB / env, never hardcoded ✓
